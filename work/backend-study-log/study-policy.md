# Backend Study Policy

## Purpose

- FastAPI / SQLAlchemy / Pydantic Settings を「読める」だけでなく、「書ける」「説明できる」状態に近づける
- サンプルアプリの実コードを教材にして、API の入口から DB 接続までの流れをつなげて理解する
- 曖昧な理解のまま先へ進まず、足元を固めながら定着させる

## Daily Study Style

- 基本は `1日30分`
- 毎日少しずつ反復する
- 学習は `読む / 書く / 説明する` の3軸で進める
- コードリーディング本編とは分けて、この場では定着と再現力を重視する

## Quiz Policy

- 小テストは基本 `2〜3問`
- 1回の出題は `1問ずつ` にする
- ユーザーが解答したら、まず `正解 / 不正解` を先に伝え、そのあとに短く解説してから次の1問に進む
- 同じ関数や処理は、1回で終わらせず、複数の角度から繰り返し出題する
- まずは `穴埋め` `役割説明` `短い記述` `流れの説明` を中心にする
- 四択は補助としてのみ使い、基本は短い記述式と穴埋め式を優先する
- いきなり全文を書かせず、重要箇所を部分的に答えさせる問題から始める
- 短くても考える必要がある問題にする

## Order Of Study

- 基本は処理の流れに沿って順番に進める
- 優先順:
  - `backend/app/main.py`
  - `backend/app/core/config.py`
  - `backend/app/core/database.py`
- `main.py` のあとに、必要に応じて `routes` や `schemas` へ広げる
- `config.py` と `database.py` は、API の入口を支える接続先として扱う
- ランダム出題は補助的に行い、基本は入口から順に積み上げる

## Mastery Rule

- 曖昧な理解のまま次へ進みすぎない
- ただし `完璧` を待ちすぎて止まりすぎない
- 次へ進む目安:
  - 自力で短く書ける
  - 短く説明できる
  - 1つ前のファイルとのつながりを言える
- 同じ論点で `2回連続ほぼ正解` を一つの目安にする
- 怪しかった論点は翌日以降に再出題する
- 1つの関数や処理について、ユーザーが「理解できた」と感じるまで次へ進まない

## Review Rule

- 新しい範囲に進んでも、前に学習した範囲を毎回少し混ぜる
- `未理解` `要再確認` `定着気味` の感覚で進捗を見る
- 進捗の見た目より、再現できる理解を優先する
- ユーザーから出題の粒度や進め方の修正依頼があったら、直ちに次回以降へ反映する
- 1つの論点は、同じ意味を別表現で何度も確認して定着を優先する

## Files To Read Before Each Quiz

- この `study-policy.md`
- `work/backend-study-log/` にあるこれまでの学習記録
- 特に直近のログと、そこで曖昧だった論点

## How To Prepare The Next Quiz

- 前回までのログを読んで、怪しかった論点を確認する
- その日の対象ファイルを1つ決める
- そのファイルの中でも、まずは最重要の1関数や1処理に絞る
- その論点を `穴埋め` `書く` `説明する` `流れをつなぐ` の順で、少しずつ難易度を上げて出題する
- 必要に応じて、前回の苦手論点を1問目に再出題する
- コーディングテスト形式が合うときは、重要箇所の穴埋めコード問題も混ぜる
